# Stack-Analysis.md

## Purpose
How to analyze the layers of systems that make a business work — the invisible infrastructure.

## The Stack Question
What layers of technology, process, or organization does this business depend on? What's broken in each layer? How do the layers interact?

## How to Build It
1. Identify 3-5 foundational stacks (from Technology.md)
2. For each stack: what it is, how it applies, what's broken, component count
3. Map stack interactions: enables, breaks, or neutral
4. Count fragmentation: how many tools/portals/logins per stack?
5. Identify technology gaps: what's missing that would unlock value?

## Stack Analysis Rules
- Name each stack clearly (not "Layer 1" — give it a real name)
- Define each stack in one sentence
- Be specific about what's broken (not "it's bad" — say "17 separate logins cause 2 hours of daily switching")
- Show interactions between stacks (where information flows, where it's blocked)

## What Good Looks Like
- 3-5 stacks identified (not 2, not 10)
- Each stack has a clear name and definition
- "What's broken" is specific, not generic
- Fragmentation counts reveal actual complexity
- Stack interactions show where the system breaks
